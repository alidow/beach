# Multi-Client WebRTC Status ‚Äì 2025-09-29

## Context
- Host (`beach-human`) and signaling server (`beach-road`) run locally; multiple browser or CLI clients can join the same session.
- Recent failures stemmed from the host repeatedly re-posting SDP offers, clobbering answers intended for earlier attempts.

## Current Status
- **Handshake IDs everywhere.** Every offer, answer, and ICE candidate now carries a `handshake_id` that the host, browser, and signaling server validate before acting on the payload.
- **Per-peer signaling buckets.** `beach-road` stores offers per `(to_peer, handshake_id)` and answers per `handshake_id`, so retries no longer overwrite each other.
- **Correct peer IDs.** `beach-human` and `beach-web` use the server-assigned peer IDs (`join_success.peer_id`) when polling or posting SDP, which keeps HTTP endpoints aligned with WebSocket traffic.
- **Stale-signal rejection.** Both sides discard ICE or SDP that references an unexpected `handshake_id`, preventing churn when a new negotiation begins.

## Testing (60‚ÄØs enforced via `test-timeout` or Vitest defaults)
- **Rust (`apps/beach-human/tests/webrtc_transport.rs`):**
  - `webrtc_signaling_end_to_end` covers a full offerer/answerer exchange and asserts that bidirectional frames deliver once the data channel opens.
  - `webrtc_multiple_handshakes_use_unique_ids` spins up three sequential negotiations against the stub signaling server to confirm each run records a distinct `handshake_id`.
- **TypeScript (`apps/beach-web/src/transport/webrtc.test.ts`):**
  - `appendParams helper` verifies that browser polling always injects `peer_id` and `handshake_id` query parameters without mutating existing values.

### Runbook

